# 4.4. Iniciativas Extras

Registro de iniciativas complementares adotadas pela equipe durante a entrega de **Arquitetura & Reutilização de Software**. Cada decisão abaixo é descrita com **motivação técnica**, **limitações reconhecidas** e **referência bibliográfica** quando aplicável — evitando descrições apenas celebratórias.

---

## Comunicação

No início do projeto, a equipe completa — **10 integrantes** — utilizou o **Discord** como canal principal. Por lá, alinhávamos divisões de trabalho e levantávamos repositórios de referência arquitetural (ex.: APIs REST com JWT, SPAs React com camada de adaptadores).

Com o avanço das entregas, restaram **3 dos 10 integrantes** (Amanda De Moura, Anna Clara Brandão e Mariana Martins). A redução do time tornou inviável manter o modelo de coordenação informal de um grupo grande; passamos a usar o **WhatsApp** para decisões síncronas e criamos um **grupo exclusivo** para organização, divisão frontend/backend e priorização de RFs.

### Embasamento e trade-offs

| Aspecto | Decisão | Fundamentação | Limitação reconhecida |
| --- | --- | --- | --- |
| Estrutura do time | 3 pessoas, 2 frentes (FE/BE) | **Lei de Conway** (Conway, 1968): a estrutura de comunicação do time tende a refletir-se na estrutura do software — separar frontend e backend exige canais explícitos de contrato (Swagger, RFs) | Com apenas 1 dev backend, gargalo em integrações; mitigado com OpenAPI e docs 4.4–4.8 |
| Canal síncrono | WhatsApp (núcleo) | Baixa latência para desbloquear dependências entre camadas | Histórico difícil de auditar — decisões relevantes foram **replicadas** em issues/PRs no GitHub |
| Canal assíncrono | GitHub (código + issues) | Fonte única de verdade versionada; rastreabilidade RF → commit exigida na entrega | Curva de aprendizado de Git/PR para quem não participou até o fim |
| Documentação | GitPages (`docs/`) | Contrato estável para avaliação; desacopla apresentação do repositório de código | Duplicação parcial com anotações Swagger — aceita para públicos distintos (docente vs. dev) |

| Fase | Canal | Público | Finalidade |
| --- | --- | --- | --- |
| Início do projeto | Discord | Grupo completo (10 pessoas) | Alinhamentos gerais, divisão inicial, referências arquiteturais |
| Continuidade da entrega | WhatsApp | Núcleo ativo (3 pessoas) | Decisões formais de organização e priorização |
| Desenvolvimento e docs | GitHub + GitPages | Equipe + avaliação | Versionamento, PRs, issues, documentação rastreável |

**Referências:** Conway (1968), *How Do Committees Invent?*; Bass, Clements & Kazman — *Software Architecture in Practice* (atributo de **modificabilidade** via comunicação explícita entre equipes).

## Documentação de reutilização

**Responsável:** Amanda De Moura

A equipe adotou um roteiro mínimo inspirado em templates de documentação arquitetural (arc42 — decisões e building blocks; IEEE Std 1016 — descrição de componentes) adaptado ao escopo acadêmico. Cada item reutilizado deve responder:

| Pergunta | O que responder | Critério técnico |
| --- | --- | --- |
| **O que faz?** | Papel da biblioteca/serviço no sistema | Identificar **qual padrão ou estilo** ela materializa (ex.: *Facade*, *Strategy*, *Repository*) |
| **Por que utilizei?** | Alternativa descartada e motivo | Comparar com pelo menos uma opção (ex.: Passport vs. auth manual; Prisma vs. SQL puro) |
| **No que ajudou no projeto?** | Efeito mensurável ou verificável | Atributo de qualidade afetado: **modificabilidade**, **segurança**, **testabilidade** (Bass et al.) |

Esse roteiro está aplicado na [visão geral](/ArquiteturaReutilizacao/backend/00.VisaoGeral.md) e nos módulos por RF ([02](/ArquiteturaReutilizacao/backend/02.Autenticacao.md)–[06](/ArquiteturaReutilizacao/backend/06.SincronizacaoGooglePlaces.md)), cruzados com os documentos por requisito ([4.4](/requisitos/RF01-backend/4.4.Autenticacao.md)–[4.8](/requisitos/RF-google-places-backend/4.8.SincronizacaoGooglePlaces.md)).

**Referências:** Gamma et al. — *Design Patterns* (1994); Bass, Clements & Kazman — *Software Architecture in Practice*; Richards & Ford — *Fundamentals of Software Architecture* (trade-off analysis).

## Fluxo de trabalho

**Responsável:** Anna Clara Brandão

Fluxo operacional do frontend, alinhado a **integração contínua leve** (sem pipeline automatizado completo, mas com gates manuais explícitos):

```text
Pega a issue → Desenvolve → Testa → Manda print aqui → Merge se for aceito
```

| Etapa | Descrição | Padrão / prática relacionada |
| --- | --- | --- |
| **Pega a issue** | Card no GitHub Projects vinculado a um RF | Rastreabilidade requisito → implementação |
| **Desenvolve** | Código no frontend reutilizando átomos/moléculas (Atomic Design) | Composição sobre duplicação (Open/Closed — Martin, SOLID) |
| **Testa** | Vitest + Testing Library; validação manual de fluxo | Pirâmide de testes — unitário antes de integrar (Fowler) |
| **Manda print aqui** | Evidência visual na issue/WhatsApp | Revisão por pares informal — substituto de code review formal em time reduzido |
| **Merge se aceito** | PR para `dev` | Gate de qualidade antes de integrar na linha compartilhada |

**Limitação:** a etapa “print” não substitui revisão de código estruturada; em produção, seria necessário checklist de PR (lint, testes CI, diff review).

## Outras decisões

### Fluxo de Desenvolvimento (Git)

Variante de **GitHub Flow** com branch de integração `dev` (intermediária entre feature branch e `main`):

```text
Criar branch a partir da dev
  → Desenvolver a funcionalidade ou correção
  → Realizar commits
  → git pull origin dev / git merge dev
  → Validar comportamento em dev
  → Abrir PR da branch de trabalho para dev
```

| Etapa | Descrição | Motivo técnico |
| --- | --- | --- |
| **Branch a partir de `dev`** | Isola trabalho por RF/issue | Reduz acoplamento entre features paralelas (modificabilidade) |
| **Commits incrementais** | Histórico atômico e reversível | Facilita `git bisect` e rollback parcial |
| **Merge `dev` antes do PR** | Rebase/merge antecipado | Detecta conflitos **antes** da revisão — evita “integration hell” |
| **Validar em `dev` local** | Smoke test pós-merge | Confirma que a feature não quebra contratos existentes (regressão) |
| **PR → `dev`** | Revisão + histórico no GitHub | Rastreabilidade exigida na entrega (participações & commits) |

**Trade-off:** não há branch `release` nem tags semver — adequado ao escopo acadêmico, insuficiente para deploy contínuo em produção sem CI/CD adicional.

**Referência:** GitHub Flow (GitHub, 2011); Fowler — *Continuous Integration* (integração frequente com build/test automatizado como evolução desejável).

Esse fluxo foi adotado no **backend** e no **frontend**, complementando o fluxo por issue da seção anterior.

## Organização no Projects

**Responsável:** Amanda De Moura

Os RFs foram modelados no **GitHub Projects** (board *EU AMO PIRI*) como **unidades de trabalho rastreáveis**, cada uma com critérios de aceitação e cenários **BDD** (Given/When/Then) na descrição da issue — alinhado a especificação executável (North, 2006; Wynne & Hellesøy).

| Coluna | Função | Papel arquitetural |
| --- | --- | --- |
| **Requisitos** | Catálogo dos RFs (RF01, RF03, RF04…) | **Viewpoint de requisitos** — o que o sistema deve fazer |
| **Backlog** | Itens não iniciados | Fila priorizada (WIP não limitado formalmente) |
| **Ready** | Definition of Ready implícita | Card só avança se critérios de aceitação estiverem definidos |
| **In progress** | WIP limitado (4 cards) | Reduz multitarefa — foco em concluir antes de puxar novo item |
| **In review** | Validação antes de Done | Gate de qualidade manual |
| **Done** | RFs entregues (`[Frontend] RF04`, etc.) | Evidência de implementação; elo com commits |

A organização no Projects sustenta a **rastreabilidade forward** (RF → issue → commit → doc) exigida na avaliação de arquitetura e reutilização.

**Limitação:** colunas não automatizam transição via CI; movimentação é manual — aceitável para equipe de 3, mas não escala sem bots/hooks.

Comprobatório: [GitHub Projects — org UnBArqDsw2026-1-Turma02](https://github.com/orgs/UnBArqDsw2026-1-Turma02/projects).

**Referências:** Beck — *Extreme Programming Explained* (historias de usuário); North — *Introducing BDD* (2006); Kniberg — Kanban e limites de WIP.
